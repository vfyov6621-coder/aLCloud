**1. Концепция: почему без сервера**

Архитектура без собственного сервера --- это десктопное или
web-приложение, которое напрямую подключается к API облачных
провайдеров. Все данные, токены и метаданные хранятся локально на
устройстве пользователя. Нет промежуточного слоя --- клиент сам
выполняет все API-вызовы, синхронизацию и индексацию файлов.

Преимущества подхода: нулевые расходы на инфраструктуру, полный контроль
над данными, отсутствие синглов одной точки отказа, максимальная
приватность. Все токены доступа хранятся исключительно на устройстве
пользователя и никогда не передаются третьим лицам. Приложение можно
распространять как Electron-апп (Windows/macOS/Linux) или как
Tauri-приложение (меньший размер, написано на Rust).

**3. Архитектура приложения**

**3.1. Структура**

Приложение состоит из двух слоёв: UI-обёртки (Electron/Tauri) и
абстракции провайдеров (provider adapter layer). Каждый провайдер
реализует единый интерфейс IProvider с методами: upload(), download(),
list(), delete(), search(). Локальный индекс файлов хранится в SQLite
или IndexedDB (для web-версии), позволяя быстро поискать и навигировать
без запросов к API.

**4. Ключевые особенности реализации**

**4.1. Авторизация провайдеров**

Для Google, Яндекс, Mail.ru и GitHub используется стандартный OAuth 2.0
flow с PKCE (потому что нет сервера для хранения client_secret).
Пользователь нажимает «Подключить» → открывается браузер с страницей
провайдера → пользователь даёт разрешение → redirect_uri перенаправляет
в приложение → токен сохраняется локально. Для Telegram используется
MTProto-подключение через номер телефона и код подтверждения.

**4.2. Синхронизация и кеширование**

При первом подключении провайдера приложение выполняет полную индексацию
файлов и сохраняет метаданные в локальную SQLite/локальную БД. Далее
работает инкрементальная синхронизация через Webhook / polling (где
поддерживается). Поиск, навигация и отображение работают быстро
благодаря локальному кешу. Миниатюры изображений кешируются на диске.

**4.3. Единый интерфейс провайдеров**

Каждый провайдер реализует класс Provider с методами: authenticate(),
list(path), upload(file, path), download(fileId), delete(fileId),
getQuota(). UI обращается к провайдерам универсально, не зная о
конкретной реализации. Пользователь видит единую файловую систему с
боковой панелью провайдеров и может переносить файлы между ними
перетаскиванием.

**5. Ограничения безсерверной модели**

1.  Отсутствие коллаборации в реальном времени --- невозможно без
    сервера синхронизировать изменения между устройствами.

2.  CORS-ограничения для web-версии --- некоторые API не поддерживают
    клиентские вызовы, требуется десктопная версия.

3.  Ограниченный объём кеша --- метаданные только для подключённых
    провайдеров, без глобального поиска.

4.  Обновление OAuth-токенов --- refresh-токены хранятся локально, при
    потере нужно переавторизация.
